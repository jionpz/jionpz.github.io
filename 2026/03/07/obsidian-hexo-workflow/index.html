Hexo 与 Obsidian 协同方案

发表于2026-03-07|更新于2026-03-08|知识管理
|浏览量:

引子

我现在同时在维护两套系统：

  • 一个 Obsidian Vault，用来记录知识、沉淀笔记、整理写作素材
  • 一个独立的 Hexo 项目，用来生成博客、预览页面和部署发布

因为两边都在处理 Markdown，所以很自然会冒出一个想法：

要不要把 Hexo 项目直接搬进 Obsidian 里统一管理？

这个想法短期看很合理，但仔细分析后，我越来越倾向于另一种方案：

不把它们硬合并，而是让 Obsidian 负责内容，Hexo 负责发布。

这篇文章想回答的，就是这个问题：

对于“知识库 + 博客发布”这种场景，怎样组织结构，才更适合长期维护？

一、为什么会想把 Hexo 和 Obsidian 放在一起

这个念头其实很常见，因为表面上看，它们处理的都是 Markdown 文件。

于是很容易得出一个直觉：

  • 笔记是 Markdown
  • 博客文章也是 Markdown
  • 那不如全部放在一个地方

这种做法在早期确实有吸引力：

  • 写作入口统一
  • 文件集中
  • 不用来回切换目录
  • 看起来更“省事”

但问题往往不是一开始出现，而是在内容逐渐增多之后才显现出来。

二、为什么我不推荐把整个 Hexo 项目搬进 Vault

我现在不太推荐把整个 Hexo 工程直接塞进 Obsidian Vault，主要有三个原因。

1. 内容层和工程层不是一回事

Obsidian 更适合做的是：

  • 知识笔记
  • 灵感记录
  • 草稿写作
  • 双链组织
  • 长期内容沉淀

Hexo 更适合做的是：

  • 页面渲染
  • 主题管理
  • 本地预览
  • 配置管理
  • 部署发布

它们都和“文章”有关，但扮演的角色并不一样。

如果把两者混在一起，后面很容易出现这些情况：

  • 笔记目录里混入工程文件
  • 博客工程里混入大量未发布内容
  • 草稿、正式文章、知识笔记的边界越来越模糊

短期方便，长期会越来越难维护。

2. Obsidian 会被工程文件污染

如果整个 Hexo 项目进入 Vault，通常会出现这些内容：

  • package.json
  • _config.yml
  • themes/
  • node_modules/
  • public/

这些文件对 Hexo 是正常的，但对 Obsidian 来说，它们不是内容，而是工程噪音。

直接带来的影响通常有：

  • 搜索结果变吵
  • 文件树变乱
  • 图谱失真
  • 阅读和写作的注意力被打断

如果你的 Vault 本来是知识库，这种污染会非常明显。

3. 草稿和正式文章容易混在一起

不分层时，最常见的问题不是技术问题，而是内容管理问题。

比如：

  • 一些只是草稿的内容被放进发布目录
  • 一些尚未整理好的笔记被当作文章维护
  • 一些原本只适合内部沉淀的内容被过早公开化

一开始可能只是“先凑合”，但后面会越来越难判断：

哪些内容是知识笔记，哪些内容是博客草稿，哪些内容已经适合发布。

三、我更推荐的方案：内容层与发布层分离

比起“全部塞进一个目录”，我现在更认同一种分工明确的结构：

  • Obsidian 是内容主库
  • Hexo 是发布工程

也就是说，两边不是竞争关系，而是协作关系。

Obsidian 负责什么

Obsidian 更适合承担这些工作：

  • 记录灵感
  • 写草稿
  • 沉淀知识
  • 整理素材
  • 规划系列文章
  • 维护模板和标签体系

Hexo 负责什么

Hexo 更适合承担这些工作：

  • 渲染页面
  • 处理 permalink
  • 管理主题表现
  • 提供本地预览
  • 构建静态页面
  • 执行部署发布

真正需要打通的，其实只有两类内容：

  • 正式文章
  • 发布所需图片资源

这就足够了。

四、我采用的目录结构

为了让这套协同关系稳定下来，我给内容层和发布层分别定义了边界。

Obsidian 侧：内容主库

这几个目录的职责分别是：

  • 00-Ideas：只放选题、灵感、提纲
  • 01-Drafts：只放还在写的草稿
  • 02-Posts：只放已经达到发布标准的正式文章
  • 04-Assets：只放博客使用的图片和附件
  • 99-Templates：统一存放模板

这套结构的核心思想是：

不要让“还在思考中的内容”和“准备发布的内容”混在一起。

Hexo 侧：发布工程

Hexo 不需要读取我的整个知识库，它只需要接触：

  • source/_posts：正式文章
  • source/img：发布图片

这样 Hexo 只面对“可发布内容”，不会误碰草稿和知识笔记。

五、一篇文章应该怎样流动

我希望以后每篇文章都遵循同一条路径：

对应到目录就是：

  1. 先在 00-Ideas 记录想法
  2. 再在 01-Drafts 写成草稿
  3. 成熟后移到 02-Posts
  4. 再同步到 Hexo 的 source/_posts
  5. 本地预览没问题后再发布

这条流程看起来多了几步，但换来的是更清晰的内容边界：

  • 想法不会丢
  • 草稿不会误发
  • 正式文章区始终干净
  • 发布工程只处理真正要发布的内容

六、为什么这套结构更适合长期维护

我现在越来越重视一件事：

工具可以换，但内容结构最好不要反复推翻。

从这个角度看，内容层和发布层分离，至少有三个明显优势。

1. 内容不会被发布工具绑死

如果以后我不再用 Hexo，而改用其他博客系统，那么：

  • Obsidian 里的目录结构依然成立
  • 草稿和正式文章的分层依然成立
  • 图片和模板体系也依然成立

换句话说：

发布工具是可替换的，内容主库才是长期资产。

2. 写作和发布可以分别优化

结构分开之后，我可以分别优化两端，而不用互相牵连。

写作层可以优化：

  • 模板
  • frontmatter 规范
  • 标签体系
  • 草稿流程
  • 双链和知识组织

发布层可以优化：

  • 主题
  • SEO
  • permalink
  • 分类页
  • 部署流程

这会比把所有问题都堆进同一个目录里清晰得多。

3. 更符合“知识沉淀到文章发布”的自然流程

并不是所有笔记都应该直接公开。

更合理的流程应该是：

  1. 先在知识库里记录和沉淀
  2. 再从中提炼出值得公开表达的主题
  3. 写成结构化文章
  4. 最后再交给博客系统发布

也就是说：

知识库不是博客，博客也不应该等于知识库。

两者相关，但不应该完全重叠。

七、这套方案适合谁，不适合谁

适合

  • 把 Obsidian 当长期知识库使用的人
  • 同时又想维护一个正式博客的人
  • 希望把“写作”和“发布”分开管理的人
  • 在意长期可维护性的人

不太适合

  • 只想快速搭一个一次性博客的人
  • 没有知识库需求，只想单纯写站点文章的人
  • 希望所有东西都尽量极简、尽量少分层的人

如果你的目标只是“尽快有个博客能发文”，那直接在 Hexo 里写也完全可以。
但如果你已经明显同时拥有：

  • 知识沉淀需求
  • 草稿管理需求
  • 文章发布需求

那分层会更合适。

八、我准备怎么落地这套方案

从执行上，我不打算一开始就做大迁移，而是准备分几步慢慢落地。

第一步：建立博客内容区

先在 Obsidian 里建立：

  • 00-Ideas
  • 01-Drafts
  • 02-Posts
  • 04-Assets
  • 99-Templates

第二步：先跑通一两篇文章

不急着整理全部历史内容，而是先拿 1~2 篇文章完整走一遍流程，确认：

  • frontmatter 是否顺手
  • 图片路径是否稳定
  • Hexo 预览是否正常

第三步：再逐步整理历史内容

等新流程稳定后，再慢慢把旧内容分流到：

  • 知识笔记
  • 博客草稿
  • 正式文章

这样做的好处是：

  • 风险低
  • 可回退
  • 不容易一次性把现有内容搞乱

总结

对于我现在这种场景，更合理的方案不是把 Obsidian 和 Hexo 硬塞进同一个目录，而是让它们各自专注在自己最擅长的事情上：

  • Obsidian 负责内容生产与沉淀
  • Hexo 负责页面生成与发布

真正需要连接的，只有正式文章和发布资源。

这种结构也许不像“全部合并”那样直观，但它更清晰、更稳定，也更适合长期维护。

如果内容只是短期管理，混在一起问题不大；
但如果目标是持续写作、持续沉淀、持续发布，那么把内容层和发布层分开，通常会是更好的选择。
